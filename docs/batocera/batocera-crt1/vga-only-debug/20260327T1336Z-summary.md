## 2026-03-27 VGA-only GL isolation pass

Question for this pass:

- Is GL on the VGA `:0` session globally broken, or is EmulationStation specifically the broken layer?

Available GL tools on-box:

- `glxinfo`
- `glxgears`
- `es2_info`
- `eglinfo`

What was tested:

1. Stock session confirmation:
   - `openbox --startup emulationstation-standalone`
   - wrapper script running
   - `emulationstation --exit-on-reboot-required --windowed` running

2. Initial GL probe with ES still effectively present:
   - `glxgears` started and stayed running
   - no stdout/stderr from `glxgears`
   - `glxinfo -B`, `es2_info`, and `eglinfo -B` still hung under timeout

3. Strict probe:
   - paused wrapper process with `SIGSTOP`
   - forcibly killed only the live ES process
   - kept `openbox` and X alive
   - launched `glxgears -info` on `DISPLAY=:0.0`

Results:

- With wrapper paused and ES gone, `glxgears -info` starts and remains alive on the same X session.
- `glxgears` still produced no stderr.
- `glxinfo -B` still did not return under timeout.
- `xset q` also did not reliably return in this degraded VGA/X state.
- `ffmpeg -f x11grab` again hung and did not yield a usable frame file for the GL client run.

Interpretation:

- The VGA GL path is not cleanly healthy, but it is **not globally dead**.
- A simple GL client can start and persist on `DISPLAY=:0.0` once EmulationStation is removed from the session.
- That shifts the main fault away from "GLX/Mesa is universally broken on VGA" and toward "EmulationStation is specifically broken on top of a partially working GL path".

Strongest conclusion from all current passes:

- non-GL X clients can draw
- direct ES is black
- software-render ES fails GLX context creation
- `glxgears` can run once ES is forcibly removed

So the remaining fault domain is EmulationStation's render path / Batocera ES build behavior on this VGA session, not a simple connector or whole-session failure.
