## 2026-03-27 VGA-only modesetting / Present isolation pass

This pass tested the two most justified Xorg-side modesetting hypotheses after the ES runtime/source investigation:

1. disable DRI3 page flipping with `Option "PageFlip" "off"`
2. disable TearFree on the active modesetting path with `Option "TearFree" "false"`

### Start state

Started from the repo-managed `batocera_vga_safe` baseline.

Baseline stock session:

- `openbox`
- `emulationstation-standalone`
- `dbus-launch`
- `emulationstation --exit-on-reboot-required --windowed`

Baseline framebuffer capture:

- black frame with cursor only

### Test 1: PageFlip off

Temporary Xorg snippet used:

```conf
Section "OutputClass"
    Identifier "Batocera VGA test amdgpu pageflip off"
    MatchDriver "amdgpu"
    Driver "modesetting"
    Option "PageFlip" "off"
EndSection

Section "OutputClass"
    Identifier "Batocera VGA test radeon pageflip off"
    MatchDriver "radeon"
    Driver "modesetting"
    Option "PageFlip" "off"
EndSection
```

Observed:

- Xorg accepted the option:
  - `Option "PageFlip" "off"`
- stock wrapper-managed ES session still came up
- framebuffer capture after reboot was still black with cursor only
- pixel distribution matched the pre-change black frame exactly
- previous `Present-flip` warnings did not appear in the sampled post-change log
- after settling, `xrandr -q` and `xset q` worked again

Interpretation:

- disabling page flipping / DRI3-style Present activity changed the log behavior
- but it did **not** make ES visible

### Test 2: TearFree false

Temporary Xorg snippet used:

```conf
Section "OutputClass"
    Identifier "Batocera VGA test amdgpu tearfree off"
    MatchDriver "amdgpu"
    Driver "modesetting"
    Option "TearFree" "false"
EndSection

Section "OutputClass"
    Identifier "Batocera VGA test radeon tearfree off"
    MatchDriver "radeon"
    Driver "modesetting"
    Option "TearFree" "false"
EndSection
```

Observed:

- Xorg logged:
  - `Option "TearFree" is not used`
- `Present-flip: queue flip during flip on CRTC 0 failed: Device or resource busy` still appeared
- framebuffer capture after reboot was still black with cursor only
- pixel distribution again matched the pre-change black frame exactly

Interpretation:

- this TearFree override was not active on the real failing path
- it did not improve the session

### Current conclusion

The modesetting / Present boundary is involved, but neither:

- `PageFlip off`, nor
- `TearFree false`

is sufficient by itself to make ES visible on this VGA session.

That means the next safest test should move back up a layer:

- an ES/media-path suppression or renderer-path adjustment justified by the libVLC-heavy differential,
- not more connector/modeline work.
