## 2026-03-27 VGA-only ES runtime/source pass

This pass focused on Batocera ES runtime assumptions and the nearest possible runtime diff against `glxgears`.

### Tooling constraint

`strace` is not installed on Batocera v42 on this host, so an actual syscall trace was not available.

Nearest substitutes used:

- Batocera source/build inspection from upstream GitHub
- binary/runtime inspection on-box
- `LD_DEBUG=libs`
- `lsof`

### Batocera build evidence

From Batocera's package definition:

- Batocera builds ES from `batocera-linux/batocera-emulationstation`
- current package pin in Batocera master was `a4833ba7da27e7a348ff72adc3d058b45f90c3b1`
- on x86/x86_64 with Xorg/libGL, Batocera sets `-DGL=ON`
- it does **not** select a special EGL-only backend for this target

From upstream `CMakeLists.txt` and renderer source:

- Desktop OpenGL builds use the GL 2.1 renderer path
- the renderer creates an SDL window with `SDL_WINDOW_OPENGL`
- it creates the GL context with `SDL_GL_CreateContext`

So on this box, ES is expected to be a desktop OpenGL / GLX-style SDL app, not a KMS/Wayland-only path.

### On-box binary/runtime evidence

The ES binary links against:

- `libSDL2`
- `libGL`
- `libGLU`
- many X11/XRandR libs
- `libvlc`

Binary string inspection found:

- `SDL_GL_CreateContext`
- `present`
- `GLES2`

### Differential runtime evidence: ES vs glxgears

`glxgears`:

- loads a minimal GLX path
- `libGL`, `libGLX`, `libX11`, `libxcb`
- starts and runs on the VGA session when ES is removed

Direct ES with stock env:

- also loads `libGL`, `libGLX`, `libX11`, `libxcb`
- but additionally pulls in a much heavier libVLC graphics stack very early
- observed plugin/library loads included:
  - `libEGL`
  - `libGLESv2`
  - VLC `libglx_plugin`
  - VLC `libgles2_plugin`
  - VLC `libegl_x11_plugin`
  - VLC `libgl_plugin`
  - VLC `libglconv_vaapi_x11_plugin`
  - VLC `libglconv_vaapi_drm_plugin`
  - VLC `libglconv_vdpau_plugin`
  - multiple `vaapi`, `vdpau`, `xcb_x11`, `xcb_window` related plugins

This is the strongest differential from a simple GL client that survives on VGA.

### Xorg-side evidence

Earlier captured Xorg output during ES runs showed repeated modesetting present-flip failures:

- `Device or resource busy`
- later also `Cannot allocate memory`

Separately, the current Xorg config on-box includes:

- `/etc/X11/xorg.conf.d/20-amdgpu.conf`
- `Option "TearFree" "true"`
- `Option "VariableRefresh" "true"`

The CRT script's own example `20-modesetting.conf` explicitly comments that:

- direct modesetting pageflips are preferred
- TearFree can force a shadow framebuffer/blit path
- DRI3 is the default when not overridden

### Strongest current interpretation

The fault boundary is now narrower than "ES uses OpenGL":

- simple GLX survives
- ES does not
- ES brings in a large secondary GL/EGL/GLES/VLC/X11 video stack that `glxgears` does not

That makes the next justified live test:

- an Xorg-side Present/DRI path mitigation, or
- a narrowly targeted ES/media-path test justified by the libVLC differential

The most justified next single test is an Xorg-side Present/DRI mitigation, because it is supported by:

- Batocera's desktop-GL build evidence
- the modesetting `Present-flip` failures on this VGA session
- the fact that `glxgears` survives while ES's heavier present/media stack does not
