## 2026-03-27 VGA-only ES runtime isolation pass

This pass focused only on EmulationStation runtime behavior on the VGA session.

### Stock launch facts

The live wrapper script launches:

```text
emulationstation --exit-on-reboot-required --windowed
```

The live stock ES environment was minimal:

- `DISPLAY=:0`
- `HOME=/userdata/system`
- `PWD=/userdata`
- `LANG=en_US.UTF-8`
- `LC_ALL=en_US.UTF-8`
- `PATH=/sbin:/usr/sbin:/bin:/usr/bin`
- `DBUS_SESSION_BUS_ADDRESS=unix:path=/tmp/dbus-...,guid=...`
- `XDG_RUNTIME_DIR=/var/run`

`es.customsargs` was empty.

### Case C: direct ES with stock env, but without `--windowed`

Direct launch used:

```text
/usr/bin/emulationstation --exit-on-reboot-required
```

Result:

- process stayed alive
- no stdout
- no stderr
- no new Xorg log lines

### Interpretation

Removing `--windowed` did **not** create an immediate behavioral change visible from logs or process state.

That narrows the problem further:

- not mainly the wrapper script
- not a simple `--windowed` vs fullscreen invocation bug
- more likely EmulationStation's renderer/runtime behavior on this VGA session

Combined with earlier passes:

- non-GL X works
- a simple GL client can run once ES is removed
- ES itself remains the black component

So the next safest step is to inspect ES-specific renderer behavior or Batocera's ES build/runtime assumptions, not general display configuration.
